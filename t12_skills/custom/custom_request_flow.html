NOTES Key Design Decisions
Step 2 — Skills injected as XML, not raw text.
_build_available_skills_xml() serialises each skill's name, description, license, compatibility, and allowed-tools into structured XML so the model can reason about which skill to pick without reading every SKILL.md upfront.
Steps 6–7 — Model may emit multiple tool calls in one turn.
The model can request EXAMPLE.md and script/sample.py in the same completion turn. Application dispatches them concurrently in _dispatch_tool_calls() and returns all results in a single tool-message batch before the next LLM call.
Step 8 — execute_code merges script + inline code.
PythonCodeInterpreterTool._execute() prepends the content of script_path to the model's code argument before forwarding to the MCP Jupyter kernel — keeping the skill's helper functions in scope.
Steps 10–11 — Stateful conversation loop.
Both the assistant message and all tool messages are appended to messages[] before the next turn, preserving full context. The loop in main() continues until the user types "exit".
